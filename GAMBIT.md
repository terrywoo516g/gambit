## 1. 项目概述

**Gambit · 国王的选择**

让多个 AI 评审团把分歧摆在用户面前，由用户做最终决断的决策辅助产品。

**核心机制**：
- 用户输入问题
- Router 判断问题类型
- 3 个不同视角的 Agent（激进派/稳健派/务实派）并行输出观点
- 分歧引擎分析共识/分歧
- 用户在"决策页"选择方向
- Captain 合成最终稿

**产品定位**：用户是国王，AI 是评审团，选择权和最终稿都属于用户。

## 3. 当前架构（V3.15）

### 技术栈

- **前端**：Next.js 14 + App Router + React + Tailwind CSS
- **后端**：Next.js API Routes（在扣子编程环境运行）
- **大模型**：通过扣子编程调用国产大模型（豆包/Kimi/DeepSeek/GLM）
- **部署**：扣子编程托管（未来 websdk 嵌入腾讯云）

### 四通路架构

```
用户输入
    ↓
Router（判断问题类型）
    ↓
┌─────────┬─────────┬─────────┬─────────┐
│ direct  │ review  │ compete │decision │
│ 直通    │ 审查    │ 比稿    │ 决策    │
└─────────┴─────────┴─────────┴─────────┘
    ↓
对应处理流程
    ↓
最终输出
```

| 通路 | 适用场景 | 处理方式 |
|------|----------|----------|
| direct | 简单问答 | 单模型直接回答 |
| review | 审稿 | 三视角审查 + 汇总 |
| compete | 比稿 | 三版本生成 + 交叉评审 |
| decision | 决策 | 三 Agent 分歧 + 局势简报 + Captain 合成 |

## 4. 核心 Prompt 设计

### 4.1 Router

**职责**：判断用户输入是"事实查询"还是"决策问题"，提取任务包。

**输出格式**：
```json
{
  "type": "fact | decision",
  "confidence": 0.72,
  "prototype": "review | solve | select | compare",
  "task_package": {
    "goal": "用户目标",
    "constraints": ["约束1", "约束2"],
    "materials": "已有材料",
    "missing": ["缺失信息"]
  }
}
```

**判断原则**：
- 不确定时倾向判为决策（宁滥勿缺）
- 只有明确的信息检索类问题才判 fact 且 confidence ≥ 0.7

### 4.2 三个 Agent 人格

**共同规则**：
- 严禁对其他视角表示赞同或妥协
- key_action 要到"动作"层（"重视营销"不合格，"大 V 种草"合格）
- 敢标低 confidence
- 总字数 500 字内

#### 激进派（agent_radical）

```
【核心人格】
你追求速度、增长、颠覆。"先跑起来再说"，不完美的快比完美的慢强。
警惕过度分析、风险厌恶、温水煮青蛙。
不怕激进但不鲁莽 —— 要"高风险高回报里 ROI 最高的"。

【关注点】
- review：稿子有没有足够锐度？太安全太妥协？
- solve：更快见效的路径？最激进的打法？慢就是死。
- select：哪个选项上限最高？
- compare：哪个带来最大跃迁？
```

#### 稳健派（agent_steady）

```
【核心人格】
你追求确定性、可持续、抗风险。"活下来才有明天"，ROI≠最终回报。
警惕速胜逻辑、爆款幻想、赌徒思维。
不反对增长，要"可控风险下的稳定增长"。

【关注点】
- review：风险点在哪？最坏情况？
- solve：最稳妥路径？怎么留退路？
- select：哪个下限最高？
- compare：哪个长期可持续？
```

#### 务实派（agent_pragmatic）

```
【核心人格】
你追求可执行、投入产出比、资源匹配。"方案再好做不出来等于零"。
警惕纸上谈兵、资源错配、理想主义。
不站激进或稳健，站"当前资源下最能落地的"。

【关注点】
- review：真能执行？资源够？
- solve：现有资源下最可行的？
- select：哪个和用户资源匹配度最高？
- compare：哪个是"真能拿下"的？
```

### 4.3 Captain 合成

**输入**：立场卡 + 核心张力 + 分歧状态 + 用户选择 + 用户想法

**输出结构**（按顺序）：

```markdown
## Tradeoff 对比矩阵
| 维度 | A: {key_action} | B: {key_action} | C: {key_action} |
|------|------|------|------|
| [张力维度 1] | ... | ... | ... |

## 核心决策建议
[一句话推荐 + 条件]

## 推荐理由
[引用 Claim ID]

## 条件式推荐
我推荐 X，因为 Y 约束下 Z 最优。如果 [条件变化]，改选 W。

## 与用户想法的冲突处理（如有）

## 风险提示（如有）

## 被放弃方向的有用碎片
```

**核心原则**：
- Tradeoff 矩阵在前，先结构化对比再下结论
- 结论带条件
- 【我的想法】优先级最高
- 不和稀泥，有立场

### 4.4 Blindspot Probe

**触发条件**：分歧引擎判定为伪共识（三方高度一致）

**输出**：
```json
{
  "core_blindspot": "核心盲点一句话",
  "why_blind": "为什么这可能是盲点",
  "extra_consideration": "盲点成立时应额外考虑什么"
}
```

## 5. 数据结构

### 5.1 Claim Card（立场卡）

```json
{
  "agent_id": "agent_radical | agent_steady | agent_pragmatic",
  "claims": [
    {
      "claim_id": "A1",
      "key_action": "核心动作变量（≤15字，动词开头）",
      "thesis": "核心主张一句话",
      "rationale": ["支持理由1", "支持理由2"],
      "assumptions": ["关键假设"],
      "risks": ["核心风险"],
      "what_would_change_my_mind": ["条件变化会改变判断"],
      "confidence": 0.72,
      "alternative": "",
      "not_now": "现在不做什么",
      "why_not_now": "为什么现在不做"
    }
  ]
}
```

**字段说明**：
- `claim_id`：A 前缀 = 激进派，B = 稳健派，C = 务实派
- `key_action`：分歧引擎主比对字段，卡片主标题
- `not_now` / `why_not_now`：V3.15 新增，强制差异化输出

### 5.2 分歧引擎输出

```json
{
  "state": "伪共识 | 强共识 | 弱共识 | 强分歧 | 少数派存在",
  "avg_similarity": 0.72,
  "trigger_blindspot": true,
  "trigger_observer_auto": false,
  "consensus_summary": "三个视角都同意..."
}
```

## 6. V3.16 待做事项

### P0：Agent Prompt 修复（紧急）

**问题**：当前 Agent 输出内容与用户问题不相关。

**测试 Case**：输入"了解国内AI大模型的优势"，输出应该是关于大模型优势的内容，而不是商业化策略。

**检查点**：
- `/api/gambit/diverge/route.ts` 中 userInput 是否正确传入 Agent Prompt
- Agent Prompt 是否以用户问题为核心
- 是否有硬编码的默认话题干扰

### P1：等待时间优化

**问题**：标注 15-30 秒，实际 1-2 分钟，用户体验差。

**方案 A（推荐）**：轻量分歧发现
- 新建 `/api/gambit/diverge-lite`
- 第一层只输出：key_action（≤15字）+ one_line_why（≤30字）+ confidence
- 目标：15-20 秒进入决策页
- 完整内容在用户选择后按需加载（`/api/gambit/expand-claim`）

**方案 B**：Streaming 流式输出
- 需要先验证扣子编程是否支持 SSE
- 如果支持，等待页可以实时展示每个 Agent 的输出

### P2：按需展开层级

**问题**：V1.19 固定三层有冗余，V3.15 单层 + "下一层"按钮很奇怪。

**方案**：层级不预判，用户按需展开
- Router 不判断需要几层
- 用户完成一层后，提示"需要更细的方案吗？"
- 点击才加载下一层

---

## 7. 关键技术决策

1. **层级不预判，用户按需展开**
   - 避免 Router 预判层数导致第一步更慢
   - 用户有控制感

2. **轻量分歧发现优先于 Streaming**
   - Streaming 实现复杂度高
   - 轻量版可以快速见效

3. **从 V1.19 保留的布局逻辑**
   - 绿色共识摘要区
   - 分歧卡压缩（高度小、默认折叠理由）
   - 返回/重新开始按钮
   - 信息层级：先共识后分歧

## 10. 跨模型异源配置

| 角色 | 模型 | 理由 |
|------|------|------|
| 激进派 Agent | Kimi moonshot-v1-32k | 长上下文 + 发散性思考 |
| 稳健派 Agent | 豆包 Doubao-pro-32k | 合规稳定 + 均衡输出 |
| 务实派 Agent | DeepSeek R1 | 推理能力强 + 实用导向 |
| Captain | 豆包 Seed-1.6-thinking | 深度思考 + 综合能力 |
| Router | 豆包 Lite-32k 或 GLM-4-flash | 轻量快速 |

**原则**：激进/稳健/务实三个模型来自三个不同厂商，最大化异源度，避免训练数据同源导致伪共识。
