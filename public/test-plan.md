# 手动测试方案

## 测试原则

不需要任何工程实现。直接在任意 AI 对话窗口（豆包、Kimi、ChatGPT、Claude）中，手动把 prompt + 测试输入贴进去跑，看输出质量。

每个 prompt 测 3-5 个 case，重点关注：输出格式是否合规（能被 JSON.parse）、内容质量是否可用、边界 case 是否正确。

---

## 测试一：Router 四分类

**目标**：验证 Router 能不能准确分流，尤其是边界 case。

**做法**：把 `prompts/router_v3.md` 的完整内容贴到对话窗口，把 `{{user_input}}` 替换成测试输入，看输出。

**测试用例**（20 条，覆盖四条通路 + 边界）：

| # | 输入 | 期望 route | 期望 complexity | 边界说明 |
|---|---|---|---|---|
| 1 | "Python 怎么读 CSV" | direct | low | 基础 |
| 2 | "可口可乐总部在哪个城市" | direct | low | 基础 |
| 3 | "用 500 字解释区块链原理" | direct | medium | 需要组织但答案方向明确 |
| 4 | "这段合同翻译有没有法律语义偏差：[贴一段合同]" | direct | high | 高复杂度直通 |
| 5 | "帮我看看这封邮件语气对不对：亲爱的张总，关于上次会议讨论的方案，我认为贵司的报价偏高..." | review | medium | 基础审查 |
| 6 | "这个活动方案有没有漏洞：[贴一段方案]" | review | medium | 审查 vs 决策边界 |
| 7 | "帮我优化这段代码的性能：def func(): ..." | review | medium | 代码审查 |
| 8 | "帮我写一个可口可乐五一推广文案" | compete | medium | 基础比稿 |
| 9 | "这三个标题哪个好：A. 夏日狂欢 B. 清凉一夏 C. 暑期特惠" | compete | medium | 有候选物的比稿 |
| 10 | "帮我写一封拒绝供应商的邮件" | compete | medium | 生成产出物 |
| 11 | "帮我写个标题" | compete | low | 简单但仍是比稿 |
| 12 | "该不该现在换工作" | decision | high | 基础决策 |
| 13 | "我们的 SaaS 产品怎么冷启动" | decision | high | 策略决策 |
| 14 | "团队 3 人，Q3 先做增长还是先做商业化" | decision | high | 资源分配 |
| 15 | "把这篇文章改成小红书版本：[贴文章]" | compete | medium | ⚠️ 边界：不是审查是比稿 |
| 16 | "帮我写三个方案然后选一个" | compete | medium | ⚠️ 边界：不是决策是比稿 |
| 17 | "这个方案有没有漏洞：[贴方案]" | review | medium | ⚠️ 边界：不是决策是审查 |
| 18 | "帮我翻译这段话：今天天气很好" | direct | low | 简单翻译 |
| 19 | "GPT-4 和 Claude 哪个更好" | decision | medium | 开放性比较 |
| 20 | "用 React 写一个 Todo 应用" | compete | medium | 代码生成 |

**评判标准**：

- route 判断正确率 ≥ 85%（17/20）为合格
- 边界 case（#15, #16, #17）全部正确为优秀
- JSON 格式合规率 100%（不合格直接不过）
- 如果某类错误集中出现（比如总把比稿分到审查），说明 prompt 的该区域 few-shot 不够

## 测试二：核心张力

**目标**：验证核心张力是否具体、不空洞。

**做法**：把 `prompts/captain_core_tension_v3.md` 贴进去，替换变量。

**测试用例**（5 条）：

| # | goal | 期望特征 |
|---|---|---|
| 1 | "该不该现在离职去创业" | 应该具体到当前情境，如"稳定月薪 vs 创业期零收入赌三年后回报" |
| 2 | "SaaS 产品冷启动策略" | 不能是"速度 vs 质量"，应该是类似"花钱买流量快速验证 vs 深耕种子用户口碑起量" |
| 3 | "团队 3 人，先做增长还是商业化" | 应该涉及具体资源约束 |
| 4 | "公司年会要不要取消改成团建" | 应该涉及员工感受、预算、文化信号等具体维度 |
| 5 | "新产品定价定高还是定低" | 不能是"利润 vs 市场"，应该更具体 |

**评判标准**：

- 0/5 使用了禁用词 → 合格
- 核心张力读完后，不了解背景的人也能大致理解用户在纠结什么 → 优秀
- 如果出现"速度 vs 质量"、"短期 vs 长期"这类万能张力 → prompt 需要加更多反面示例

## 测试三：三个 Agent（决策通路）

**目标**：验证三张立场卡是否一眼可区分、key_action 是否到"动作层"。

**做法**：先用核心张力 prompt 跑出张力，再分别用三个 Agent prompt 跑。

**测试用例**（3 条决策问题，每条跑 3 个 Agent = 9 次调用）：

| # | goal | 关注点 |
|---|---|---|
| 1 | "该不该现在离职去创业，我有 50 万存款，老婆怀孕了" | 三个 key_action 是否完全不同方向 |
| 2 | "我们是一个 5 人小团队做的 AI 写作工具，DAU 500，要不要接受一个大厂的收购 offer" | rationale 第二条是否提到"为什么另外两个方向不对" |
| 3 | "公司要扩张，开新城市还是深耕现有城市" | thesis 是否说清"先做什么、不先做什么、为什么" |

**评判标准**：

- 三张卡的 key_action 放在一起看，3 秒内能区分三个方向 → 合格
- key_action 长度 8-18 字，到动作层（不是"重视 XX"）→ 合格
- rationale[1] 提到了另外两个方向为什么不优先 → 合格
- thesis 包含"先做/不先做/为什么"三件事 → 合格
- JSON 格式合规 → 必须

## 测试四：局势简报

**目标**：验证局势简报是否是"人话"、速读总结是否抓住重点。

**做法**：用测试三产出的三张立场卡作为输入，贴到 `prompts/briefing_v3.md` 中跑。

**测试用例**：用测试三的 3 个 case 的 Agent 输出。

**评判标准**：

- common_ground 不是废话（"三方都同意目标很重要"不合格）→ 合格
- key_differences 每个维度下三方的一句话立场互不相同且具体 → 合格
- king_question 不是用户原始问题的复述，而是深入一层 → 合格
- quick_take 40-60 字 → 合格
- 0 个禁用词 → 合格

---

## 测试五：比稿交叉评审

**目标**：验证交叉评审能不能有效区分三个版本。

**做法**：
1. 在三个不同 AI（Kimi、智谱、DeepSeek）上分别输入同一个创作需求，拿到三个版本
2. 把三个版本贴到 `prompts/compete_review_v3.md` 中跑

**测试用例**（3 条）：

| # | 创作需求 | 关注点 |
|---|---|---|
| 1 | "帮我写一个可口可乐五一推广文案，面向 18-25 岁年轻人，投放小红书" | label 是否一眼区分风格 |
| 2 | "帮我写一封婉拒合作邀请的邮件，对方是我们重要合作伙伴" | key_differences 维度是否贴合任务 |
| 3 | "帮我写一个 App 上线发布会的活动方案" | recommended_base 的理由是否说服人 |

**评判标准**：

- 三个 version 的 label 互不相同且具体（不是"版本A比较好"）→ 合格
- top_strength / top_weakness 具体到可引用原文 → 合格
- key_differences 维度与任务类型相关（文案看"情绪/信息密度/号召力"，方案看"创新度/可执行性"）→ 合格
- recommended_base 理由清晰 → 合格

## 测试六：审查三视角 + 汇总

**目标**：验证三个审查视角是否产出了不同且有价值的 issue，汇总是否有效去重。

**做法**：
1. 准备一份有明显问题的内容（比如一封带问题的商务邮件、一份有逻辑漏洞的方案）
2. 分别用三个审查 prompt 跑出三份审查
3. 把三份审查结果贴到汇总 prompt 中跑

**测试用例**（3 条）：

| # | 被审查内容 | 关注点 |
|---|---|---|
| 1 | 一封语气偏强硬的催款邮件（你自己写一封有问题的） | 逻辑视角和受众视角是否发现不同问题 |
| 2 | 一份有数据引用错误的推广方案（故意放一些逻辑漏洞） | 逻辑视角能否抓住数据问题 |
| 3 | 一份正式的汇报稿（质量相对较高） | 是否不会过度挑刺，issue 数量是否合理（不凑数） |

**评判标准**：

- 三个视角的 issue 有明显差异化（不是三个视角发现完全相同的问题）→ 合格
- 每个 issue 的 evidence 字段确实引用了原文 → 合格
- severity 标注合理（不是全标 high）→ 合格
- 汇总的 consensus_issues 确实是多视角发现的同一问题（不是硬合并不相关的 issue）→ 合格
- 汇总没有添加新发现 → 合格
- overall_verdict 一句话准确概括 → 合格

---

## 测试七：比稿 Captain 合成

**目标**：验证合成版是否风格统一、来源标注是否到位。

**做法**：用测试五的输出，选一个底稿，贴到 `prompts/compete_captain_v3.md` 中跑。

**评判标准**：

- 最终版读起来风格统一，不像拼凑 → 合格
- 来源标注存在且合理（不是每句话都标注，也不是完全没标注）→ 合格
- 最终版可直接使用，不是分析报告 → 合格
- 改进说明在 3-5 句话以内 → 合格

---

## 测试执行建议

**优先级**：Router（测试一）> Agent + 局势简报（测试三+四）> 审查（测试六）> 比稿（测试五+七）> 核心张力（测试二）

**为什么这个优先级**：
Router 是所有通路的入口，它错了后面全白费。Agent + 局势简报是对 V2.1 的直接优化（P0 任务）。审查和比稿是新通路，可以稍后迭代。

**时间估算**：
- 测试一（Router）：30 分钟，20 个 case 手动跑
- 测试二（核心张力）：15 分钟，5 个 case
- 测试三（Agent）：30 分钟，3 个问题 × 3 个 Agent
- 测试四（局势简报）：15 分钟，复用测试三的输出
- 测试五（比稿评审）：30 分钟，需要先在三个 AI 上分别跑
- 测试六（审查）：40 分钟，准备素材 + 三个视角 + 汇总
- 测试七（比稿合成）：15 分钟，复用测试五的输出

**总计约 3 小时**。建议第一轮只跑测试一 + 测试三 + 测试四（约 1.5 小时），根据结果调 prompt，再跑其余测试。

---

## 发现问题后的处理流程

1. **Router 分错类** → 在 few-shot 区域补充该 case 作为示例，重跑验证
2. **核心张力太空洞** → 在禁用词列表中追加这次出现的空话词，在好的示例中追加一条
3. **Agent 的 key_action 太模糊** → 在 prompt 中追加一条不合格示例（"XX 不合格，YY 合格"）
4. **局势简报的 common_ground 是废话** → 在 prompt 中追加："如果你写出来的共识放在任何问题里都成立，那就不是共识，重写"
5. **审查视角高度重叠** → 检查三个审查 prompt 的维度定义是否区分度够大，必要时收窄每个视角的关注范围
6. **JSON 格式不合规** → 在 prompt 末尾追加："再次强调：直接输出 JSON，不要任何包裹或解释文字。第一个字符必须是 {"
7. **汇总添加了新发现** → 在汇总 prompt 的 "【重要】" 部分加粗并加示例说明什么叫"不添加新发现"

每次修改 prompt 后，至少用导致问题的那个 case 重新跑一遍验证修复有效，再跑 2 个其他 case 验证没有引入新问题。
