# Gambit V3 产品架构说明

## 一、核心定位重述

**Gambit · 国王的选择**
**重要决定，不该只听一个 AI 的**

Gambit 解决的是一个底层问题：**用户不信任单一 AI 的输出**。不信任的原因有三种——创作只跑一次不甘心、信息只有一个来源不敢信、思路只有一个方向怕遗漏。Gambit 用多 AI 并行加结构化比对，帮用户建立对 AI 输出的信心。

国内市场在这件事上基本空白。Genspark Mixture-of-Agents 是目前做得最好的同类产品，但它是 25 美元/月的付费功能，且国内无法使用。Poe、ChatBox 的多模型只是简单并排，没有交叉分析和合成。Gambit 在国内免费提供这个级别的多AI比对体验，这本身就是足够强的产品卖点。

## 二、用户入口设计

### 前端只有一个输入框

用户不需要选模式。打字、贴文档、放链接，扔进去就跑。Router 根据输入内容自动判断走哪条通路。

### Router 四分类逻辑

Router 判断依据是两个维度的交叉：用户要的是"产出物"还是"判断"？用户有没有带现成素材进来？

| | 要产出物 | 要判断 |
|---|---|---|
| **有素材** | 审查通路 | 决策通路 |
| **无素材** | 比稿通路 | 决策通路或直通通路 |

具体判断规则按优先级排列：

第一，用户输入是一个简单问题，有明确答案，不需要多 AI 比对（"Python 怎么读 CSV"、"可口可乐总部在哪"）→ **直通通路**。

第二，用户带了一份或多份现成内容进来，主要诉求是检查、挑刺、验证（"帮我看看这封邮件语气对不对"、"这个方案有没有漏洞"、"这三个来源的信息哪个靠谱"）→ **审查通路**。

第三，用户要求生成一个具体的产出物，没有带现成素材或只给了一个命题（"帮我写一个可口可乐五一推广文案"、"帮我写一封拒绝加班的邮件"）→ **比稿通路**。

第四，用户提出一个开放性问题，需要权衡取舍，没有标准答案（"这个业务怎么冷启动"、"该不该现在换工作"、"国产模型和海外模型差距怎么测"）→ **决策通路**。

Router 拿不准时默认走决策通路——宁可多给分析，不能漏掉有价值的分歧。

### Router 输出 Schema

```typescript
type RouterOutput = {
  route: 'direct' | 'review' | 'compete' | 'decision';
  confidence: number;        // 0.0-1.0
  task_summary: string;      // 一句话概括用户要干什么
  has_material: boolean;     // 用户是否带了素材
  task_package: {
    goal: string;
    constraints: string[];
    materials: string;       // 用户原文/附件内容
    missing: string[];
  };
};
```

## 三、四条通路详解

### 3.1 直通通路（Direct）

**适用场景**：简单问题、翻译、格式转换等不需要多 AI 比对的任务。

**用户感知**：问了就答，最快最轻。

**数据流**：

```
用户输入
  → Router 判断 route=direct
  → 3 个模型并行各跑一遍（用各自默认 prompt，不加人格）
  → Captain 自动合成一版最优回答
  → 展示最终回答（可折叠查看三个原始回答）
```

**处理节点**：

| 节点 | 说明 |
|---|---|
| 模型 A/B/C 并行 | 同一个通用 prompt，三个不同厂商模型各跑一遍，不加人格设定 |
| Captain 合成 | 交叉比对三个回答，择优合成一版。标注"三个来源一致"或"存在分歧"的部分 |

**前端展示**：

顶部：用户问题。中部：合成后的最终回答。底部折叠区："查看三个 AI 的原始回答"，展开后并排显示三个模型的输出，有差异的部分高亮。

**不使用的组件**：分歧引擎、Blindspot、Observer、局势简报。

**Agent Prompt 要点**：无人格设定，就是"认真回答这个问题"。三个模型自然差异即可。

---

### 3.2 比稿通路（Compete）

**适用场景**：用户要求生成一个具体产出物（文案、邮件、方案、代码），想要多个版本比较后择优或缝合。

**用户感知**：三个 AI 各写一版 → 看到每版的优劣分析 → 拿到一个合成最优版 → 可以微调。

**数据流**：

```
用户输入（命题 + 要求）
  → Router 判断 route=compete
  → 3 个模型并行各生成一版作品
  → 交叉评审节点：每版的优势/劣势/适用场景
  → Captain 合成最终版（标注每段来源）
  → 展示最终版 + 交叉评审摘要 + 三个原版
```

**处理节点**：

| 节点 | 输入 | 输出 | 模型 |
|---|---|---|---|
| 生成 A | task_package | 完整作品 A | 厂商 1 |
| 生成 B | task_package | 完整作品 B | 厂商 2 |
| 生成 C | task_package | 完整作品 C | 厂商 3 |
| 交叉评审 | 三份作品 + task_package | 评审报告（每版优劣 + 关键差异维度） | 独立模型 |
| Captain 合成 | 三份作品 + 评审报告 + task_package | 最终合成版 + 来源标注 | Captain 模型 |

**前端展示**：

顶部：用户命题 + 交叉评审摘要（"版本 A 口号最有传播力，版本 B 策略框架最完整，版本 C 视频脚本最落地"，2-3 句话）。

中部：合成最终版。关键段落旁标注"来自版本 A"、"综合 B+C"等。底部有"复制全文"和"导出"。

下方折叠区："查看三个原始版本 + 详细评审"。展开后三栏并排，每栏顶部是该版本的一句话特点标签（如"情绪冲击型"、"策略完整型"、"年轻化表达型"），下方是完整作品。

**交叉评审 Prompt 要点**：

不需要激进/稳健/务实人格。评审关注的维度是：创意独特性、目标受众匹配度、可直接使用程度、风格调性、结构完整性。每个版本用一句话概括核心特点，再指出最突出的优势和最明显的短板。最后列出 2-3 个"关键差异维度"，在每个维度上比较三个版本（类似 Tradeoff 矩阵）。

**Captain 合成 Prompt 要点**：

从三个版本中取各自最强的部分缝合成最终版。每个关键段落标注来源。如果三个版本在某个部分质量相当，选最适合用户命题调性的那个。合成后的作品必须可以直接使用，不是分析报告。

**不使用的组件**：分歧引擎、五态判定、Blindspot、Observer。

---

### 3.3 审查通路（Review）

**适用场景**：用户带了现成内容进来，想让多个 AI 从不同角度检查（挑刺、验证、找漏洞）。包括两种子场景——用户自己写的东西要检查，或者用户搜到的信息要交叉验证。

**用户感知**：把东西扔进来 → 三个 AI 各自挑刺 → 看到问题汇总和可信度判断 → 拿到修改建议或验证结论。

**数据流**：

```
用户输入（素材 + "帮我看看"）
  → Router 判断 route=review
  → 3 个模型并行审查同一份素材
  → 审查汇总节点：提炼共同发现 + 各自独特发现 + 矛盾点
  → Captain 合成审查报告 + 修改建议/可信度判断
  → 展示审查结论 + 问题清单 + 三份原始审查
```

**处理节点**：

| 节点 | 输入 | 输出 | 模型 |
|---|---|---|---|
| 审查 A（逻辑视角） | 素材 + task_package | 从逻辑和论证角度的审查意见 | 厂商 1 |
| 审查 B（受众视角） | 素材 + task_package | 从目标受众/使用者角度的审查意见 | 厂商 2 |
| 审查 C（风险视角） | 素材 + task_package | 从风险和极端情况角度的审查意见 | 厂商 3 |
| 审查汇总 | 三份审查意见 + 素材 | 共同发现 + 各自独特发现 + 互相矛盾的判断 | 独立模型 |
| Captain 合成 | 汇总 + 三份审查 + 素材 + task_package | 审查结论 + 修改建议 或 可信度报告 | Captain 模型 |

**前端展示**：

顶部：一句话审查结论（"三个审查视角都认为整体方向没问题，但在 X 和 Y 两个点上发现了需要注意的问题"）。

中部：问题清单，按严重程度排序。每个问题标注"三个视角都发现"（高可信）或"仅 A 视角发现"（供参考）。如果是信息验证场景，改为可信度判断——"三个来源一致认为 X 是正确的"、"关于 Y 存在分歧，A 说是、B 说否、C 不确定"。

下方：修改建议或行动建议。

折叠区："查看三份完整审查"，展开后三栏并排。

**审查 Agent Prompt 要点**：

三个审查视角不是激进/稳健/务实，而是**逻辑视角**（论证是否严密、有无逻辑漏洞、数据是否支持结论）、**受众视角**（目标读者/用户会怎么理解、有无误解风险、语气是否合适）、**风险视角**（最坏情况是什么、有无法律/合规/声誉风险、遗漏了什么极端场景）。

如果是信息验证场景（用户搜到的内容要交叉核验），三个视角调整为：**事实核验**（陈述是否准确、数据是否可查证）、**来源评估**（信息来源是否可靠、有无利益冲突）、**完整性检查**（是否遗漏了重要上下文、有无选择性引用）。

Router 在判断 route=review 的同时，需要判断子类型：review_content（审查用户自己的内容）或 review_info（验证外部信息），以便使用不同的审查视角 prompt。

**不使用的组件**：分歧引擎的五态判定（但可以复用相似度计算来量化三份审查的一致程度）、Blindspot、Observer。

---

### 3.4 决策通路（Decision）

**适用场景**：用户提出开放性问题，需要权衡取舍，没有标准答案。这是 V2.1 现有的核心流程，保留并优化。

**用户感知**：提出问题 → 看到共识和分歧 → 在关键分歧上做判断 → 拿到条件式推荐。

**数据流**：

```
用户输入（开放性问题）
  → Router 判断 route=decision
  → Captain 核心张力
  → 3 个 Agent 并行（激进/稳健/务实）
  → 分歧引擎 HTTP API → 五态判定
  → Blindspot（伪共识时触发）
  → 局势简报节点
  → 暂停，等待用户交互
  → 用户选择/填想法/点 Captain
  → Captain 合成最终稿
  → （可选）Observer
```

**处理节点**：

沿用 V2.1 现有节点，新增局势简报节点，优化以下 prompt：

| 节点 | 变更 | 说明 |
|---|---|---|
| Captain 核心张力 | 替换 prompt | 加自检环节 + 禁用空话列表，确保输出是真正的 tradeoff 而不是同义词罗列 |
| 激进派 Agent | 替换 prompt | thesis 必须说清"先做什么/暂不做什么/为什么"，rationale 第二条必须对比另外两条路 |
| 稳健派 Agent | 替换 prompt | 同上结构，人格和关注点不变 |
| 务实派 Agent | 替换 prompt | 同上结构，人格和关注点不变 |
| 局势简报（新增） | 新节点 | 在分歧引擎之后、用户交互之前，提炼共识 + 分歧 + 国王要回答的问题 |
| 分歧引擎 | 不变 | 五态判定逻辑不变 |
| Blindspot | 不变 | 伪共识时触发，逻辑不变 |
| Captain 合成 | 暂不改 | 先把审阅台体验做对，合成后续再优化 |
| Observer | 不变 | 强分歧/少数派时触发或手动触发 |

**前端展示**：

顶部：核心张力（一句话）+ 局势简报的 quick_take（速读总结）。

共识区：局势简报的 common_ground（1-2 条，"三位评审都认为 ..."）。

分歧区：局势简报的 key_differences（2-3 个维度，每个维度下三个视角各一句话的对比）。这是用户做判断的核心信息。

中部：三张立场卡（激进/稳健/务实），保留但优化。key_action 改为 8-18 字的具体动作，thesis 包含"先做什么/暂不做什么/为什么"。卡片是分歧区的展开详情，不是独立的阅读入口。

底部："我的想法"输入框 + "生成最终稿"按钮 + "让 Captain 决定"按钮。

右栏：最终稿展示区，结构沿用 V2.1（Tradeoff 矩阵 + 条件式推荐 + 放弃碎片）。

**局势简报输出 Schema**：

```typescript
type BriefingOutput = {
  common_ground: string[];     // 1-2 条真正的共识
  key_differences: {
    dimension: string;          // 短维度名，如"先验证什么"
    radical: string;            // 激进派在这个维度的立场，一句话
    steady: string;
    pragmatic: string;
  }[];                          // 2-3 个关键分歧维度
  king_question: string;        // "国王现在真正要回答的问题"
  quick_take: string;           // 40-60 字速读总结
};
```

## 五、模型分配方案

### 三模型并行层

沿用 V2.1 的跨厂商异源原则。三个槽位固定分配给三个不同厂商，无论走哪条通路都是同样的三个模型。

| 槽位 | 决策通路角色 | 其他通路角色 | 模型 | 厂商 |
|---|---|---|---|---|
| 模型 A | 激进派 Agent | 生成版本A / 审查视角A / 回答A | kimi-k2-5-260127 | Moonshot |
| 模型 B | 稳健派 Agent | 生成版本B / 审查视角B / 回答B | glm-5-0-260211 | 智谱 |
| 模型 C | 务实派 Agent | 生成版本C / 审查视角C / 回答C | deepseek-v3-2-251201 | DeepSeek |

### 其他节点

| 节点 | 模型 | 厂商 | 备注 |
|---|---|---|---|
| Router | doubao-seed-2-0-lite-260215 | Doubao | 轻量模型，四分类判断 |
| Captain 核心张力 | deepseek-r1-250528 | DeepSeek | 仅决策通路使用 |
| 交叉评审/审查汇总/局势简报 | glm-5-turbo-260316 | 智谱 | 中间分析层，用 turbo 控成本 |
| Captain 合成 | deepseek-r1-250528 | DeepSeek | 四条通路共用，prompt 不同 |
| Blindspot | glm-5-turbo-260316 | 智谱 | 仅决策通路伪共识时触发 |
| Observer | qwen-3-5-plus-260215 | 阿里 | 仅决策通路，手动或自动触发 |

## 六、前端统一展示框架

四条通路共享同一个页面结构，只是各区块填充的内容不同。

### 页面骨架

```
┌────────────────────────────────────────────────┐
│  顶部摘要区                                      │
│  （2-3 句话，用户扫一眼就知道结论）              │
├────────────────────────────────────────────────┤
│  核心内容区                                      │
│  （每条通路展示不同内容，见下表）                │
├────────────────────────────────────────────────┤
│  操作栏                                          │
│  （复制全文 / 导出 MD / 重新开始）              │
├────────────────────────────────────────────────┤
│  折叠区："查看 AI 原始输出"                     │
│  （展开后三栏并排显示原始回答/作品/审查）       │
└────────────────────────────────────────────────┘
```

### 各通路内容填充

| 区块 | 直通 | 比稿 | 审查 | 决策 |
|---|---|---|---|---|
| **顶部摘要** | 无（直接显示回答） | 交叉评审摘要：每版核心特点 + 关键差异 | 审查结论：一致发现 + 分歧点 | 局势简报：共识 + 分歧维度 + 国王要回答的问题 |
| **核心内容** | 合成后的最终回答 | 合成最终版（关键段落标注来源） | 问题清单（按严重程度排序，标注可信度） + 修改建议 | 三张立场卡 + "我的想法"输入框 + 合成按钮 → 最终稿 |
| **操作栏** | 复制 / 重新开始 | 复制 / 导出 / 重新开始 | 复制 / 导出 / 重新开始 | 复制 / 导出 / 重新开始 / 进入第 2 层 |
| **折叠区** | 三个原始回答，差异高亮 | 三个原始版本 + 每版特点标签 + 详细评审 | 三份完整审查意见 | 评审团原始观点（已有） |

### 决策通路特殊交互

决策通路保留 V2.1 的两阶段交互：阶段 1 自动执行到局势简报输出后暂停，等用户在审阅台做选择或填想法；阶段 2 用户触发 Captain 合成。其他三条通路都是全自动执行，不需要用户中间做选择。

## 七、实现路径

### 阶段一：决策通路优化（基于 V2.1 现有代码）

改动范围最小，不新增通路，只优化现有决策通路的输出质量。

**具体动作**：替换 Captain 核心张力 prompt（加自检 + 禁空话）。替换三个 Agent prompt（thesis 三件事结构 + rationale 对比要求 + key_action 8-18 字）。新增局势简报节点（在分歧引擎之后、用户交互之前）。前端审阅台顶部从"核心张力 + consensus_summary"改为"核心张力 + 局势简报四字段"。

**验证标准**：用 5 个真实决策问题跑端到端，观察三张卡的 key_action 是否一眼可区分、局势简报的共识和分歧是否准确、用户是否能在不通读三张卡的情况下理解自己在选什么。

**工作量预估**：prompt 修改 + 新增一个 LLM 节点 + 前端摘要区改版。扣子编程内完成，不涉及分歧引擎改动。

### 阶段二：直通通路 + 比稿通路

**具体动作**：Router prompt 从 fact/decision 二分升级为四分类。新增直通通路（最简单：三模型并行 + Captain 合成，无中间层）。新增比稿通路（三模型并行生成 + 交叉评审节点 + Captain 缝合合成）。前端新增比稿和直通的结果展示布局。

**验证标准**：用"帮我写一个可口可乐五一推广文案"跑比稿通路，合成结果质量不低于 Genspark Mixture-of-Agents。用"Python 怎么读 CSV"跑直通通路，体验和普通 AI 对话一致但底部能展开看三个来源。

**工作量预估**：Router prompt 重写。比稿通路需要新写交叉评审 prompt 和比稿合成 prompt。直通通路几乎无额外 prompt 工作，Captain 合成 prompt 稍作调整。前端新增两种结果布局，但骨架结构一致。

### 阶段三：审查通路

**具体动作**：新增审查通路的三种审查视角 prompt（逻辑/受众/风险 或 事实/来源/完整性）。新增审查汇总节点 prompt。Router 补充审查子类型判断（review_content vs review_info）。前端新增审查结果的问题清单展示。

**验证标准**：用户贴一篇文章说"帮我看看"，输出的问题清单按严重程度排序，每个问题标注了几个视角发现。用户搜到的信息贴进来说"这个靠谱吗"，输出可信度判断，明确标注一致和分歧。

**工作量预估**：主要是 prompt 设计工作。前端的问题清单展示是新组件但不复杂。

### 阶段四：打磨与串联

全部通路跑通后的整体优化。Router 判断准确率调优（补充 few-shot 示例）。四条通路之间的体验一致性检查。Captain 合成 prompt 统一优化。移动端基础适配。

## 八、与 V2.1 的架构差异

| 项目 | V2.1 | V3 |
|---|---|---|
| 用户入口 | 四个 Tab（求解/审稿/比较/选案） | 统一输入框，Router 自动分流 |
| Router 分类 | fact / decision 二分 | direct / compete / review / decision 四分 |
| 支持的场景 | 仅决策（求解+审稿，比较/选案待开放） | 决策 + 比稿 + 审查 + 直通 |
| Agent 人格 | 所有场景都用激进/稳健/务实 | 仅决策通路用激进/稳健/务实；比稿用风格差异；审查用视角差异 |
| 分歧引擎 | 所有场景都经过 | 仅决策通路使用 |
| 中间分析层 | 无（consensus_summary 模板拼接） | 各通路有专属中间节点（交叉评审/审查汇总/局势简报） |
| 前端布局 | 三联布局固定 | 统一骨架 + 通路专属内容填充 |
| 用户交互断点 | 仅决策通路有（选卡 + 填想法） | 仅决策通路有，其他三条全自动 |

## 九、不做的事（明确边界）

与 V2.1 一致的不做项继续保留。额外明确以下几点：

不做通路之间的动态切换（用户进了比稿通路，不能中途跳到决策通路。要重新开始）。不做 Agent 数量扩展（每条通路固定三个模型）。不做对话式追问（每条通路都是一次输入一次输出，不是多轮对话）。不做用户自选模型（三个模型固定分配，用户不可更换）。不做实时联网搜索（审查通路的信息验证基于模型自身知识，不调搜索 API）。
